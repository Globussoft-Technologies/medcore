## What depends on Jitsi server configuration

The API layer assumes the client browser reaches a Jitsi server at
`JITSI_DOMAIN` (default `meet.jit.si`). For production, operators must:

1. Set `JITSI_APP_ID` + `JITSI_APP_SECRET` (signed JWT) if using JaaS or a
   self-hosted prosody-with-JWT plugin.
2. For recording, install **Jibri** alongside the Videobridge and allow the
   moderator JWT feature `recording=true`. Jibri writes mp4 files to a local
   path or S3; operators must push the resulting URL back to
   `POST /api/v1/telemedicine/:id/recording/stop` via a finalize webhook.
3. For moderated "lobby / waiting-room" behaviour, enable the
   `muc_lobby_rooms` prosody module. The API `waitingRoomState` column is the
   source-of-truth for the **business** admit/deny flow ‚Äî the Jitsi lobby is
   purely the UX enforcement.

These are all server-side concerns; the MedCore API degrades gracefully to
plain `meet.jit.si` rooms when none of the above is configured.
