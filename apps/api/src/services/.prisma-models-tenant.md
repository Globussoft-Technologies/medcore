# Multi-Tenant Migration Plan

Status: **DESIGN ONLY — no schema changes applied.** This document captures
the target architecture for converting MedCore from a single-tenant
deployment to a per-hospital tenant-isolated SaaS. Do **not** edit the
Prisma schema based on this file until the full plan has been reviewed and
a staged migration PR is ready.

## Goals

1. Every top-level business record is owned by exactly one `Tenant` (a
   hospital / clinic chain / franchise).
2. A single MedCore instance can serve many tenants; a request for tenant A
   must never see tenant B's rows.
3. Enforcement is defense-in-depth: middleware + Prisma middleware + optional
   Postgres Row-Level Security.

## New model: `Tenant`

```prisma
model Tenant {
  id        String   @id @default(uuid())
  name      String
  subdomain String   @unique          // e.g. "stmarys" → stmarys.medcore.app
  plan      String   @default("TRIAL") // TRIAL | STARTER | PRO | ENTERPRISE
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // back-relations to every tenant-scoped model (added incrementally as
  // each model is migrated)
  @@map("tenants")
}
```

## Enforcement layers

### Layer 1 — Request middleware (already scaffolded)

- `apps/api/src/middleware/tenant.ts` resolves `req.tenantId` from JWT or
  `X-Tenant-Id` header.
- `apps/api/src/services/tenant-context.ts` exposes an AsyncLocalStorage
  (`tenantAsyncStorage`) so the tenant id is reachable from any async
  descendant without threading it through function signatures.

### Layer 2 — Prisma middleware

Registered once during Prisma client construction:

```ts
prisma.$use(async (params, next) => {
  const tenantId = getTenantId();
  if (!tenantId || !TENANT_SCOPED_MODELS.has(params.model!)) {
    return next(params);
  }

  if (params.action === "create" || params.action === "createMany") {
    params.args.data = { ...params.args.data, tenantId };
  } else if (
    params.action === "findFirst" ||
    params.action === "findMany" ||
    params.action === "findUnique" ||
    params.action === "update" ||
    params.action === "updateMany" ||
    params.action === "delete" ||
    params.action === "deleteMany" ||
    params.action === "count" ||
    params.action === "aggregate"
  ) {
    params.args = params.args ?? {};
    params.args.where = { ...(params.args.where ?? {}), tenantId };
  }

  return next(params);
});
```

### Layer 3 — Postgres RLS (optional, enterprise tier)

For ENTERPRISE-plan tenants, enable Postgres Row-Level Security as a
last-resort guard in case application code forgets to use Prisma (raw SQL,
cron jobs, migrations, etc.). Policy template:

```sql
ALTER TABLE "patients" ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON "patients"
  USING ("tenantId" = current_setting('app.tenant_id')::text);
```

Prisma client would set `app.tenant_id` via `SET LOCAL` on every query
wrapped in a transaction — adds ~0.3ms/query, acceptable for enterprise.

## Migration strategy — multi-week, phased

1. **Week 1.** Ship `Tenant` table + backfill a single `default` tenant.
   No FKs added yet. No behavior change.
2. **Week 2.** Add nullable `tenantId` to each scoped model. Backfill with
   the default tenant id. Keep Prisma middleware OFF.
3. **Week 3.** Make `tenantId` `NOT NULL` on migrated models. Turn Prisma
   middleware ON in staging and run the test suite with multiple tenants.
4. **Week 4.** Production rollout behind a feature flag `MULTI_TENANT_ENABLED`.
   Rollback path: flip flag off, middleware becomes pass-through.
5. **Week 5+.** Per-tenant Postgres schemas or RLS for enterprise tier;
   per-tenant S3 prefixes for uploads; per-tenant rate limits.

## Do NOT

- Do **not** add `tenantId` to models in a single commit — Prisma
  `migrate deploy` would lock every large table during the ALTER.
- Do **not** add a FK to `Tenant` and `NOT NULL` in the same migration.
  Split into (a) nullable column + index, (b) backfill job, (c) FK + NOT
  NULL, each in its own migration.
- Do **not** share the `Medicine` or `LabTest` catalogs across tenants in
  v1 even though it would save storage — divergent formularies are the #1
  customer request from pilot hospitals.
- Do **not** attempt to backfill `Patient` across tenants. A patient at
  hospital A is a different record than "the same person" at hospital B,
  even when they share a government ID. Linking those is a separate
  Master Patient Index (MPI) problem.

## Open questions

- **Cross-tenant super-admins.** Support engineers need read access to all
  tenants. Probably a separate `admin_tenantId = "*"` JWT claim that the
  Prisma middleware honors by skipping the `where` injection.
- **Soft-delete of a tenant.** Archive to cold storage or keep forever?
- **Billing signals.** When does `plan` transition happen, and which
  Prisma events trigger usage metering?
